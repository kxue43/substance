# Refine the kxue43-pr-review Claude Skill

**IMPORTANT: Claude Code is absolutely NOT allowed to read or write any files outside of the CWD!**
**The three Claude Code skill files under discussion below are all located in the `.claude/` folder of CWD.**
**They are symlinked to from the user's home directory, so updating them is sufficient to update the skill files for the user.**

I recently decided to adopt a spec-based, narrowly scoped, session by session workflow pattern
when using Claude Code CLI.

Every session should begin with a narrow scope defined by a spec file. The research or implementation
process might involve several back-and-forth conversation with the human user, but the scope
of the session should stay focused and not expand at all. After the human user gets genuine understanding
of a research topic, he may ask Claude Code to summarize the findings into a Markdown note for future reference.
After the human user approves the implementation _of the current session_, he may ask Claude Code
to write the spec file for the next session. Then, the human user will use the `/clear` command
to clear the current context windown, start a new session, load the relevant spec and/or technical note
files, and start a new.

The key idea is to prevent the well-known "lost in the middle" attention drifting problem with LLM.

**To make the workflow smoother, I need several Claude Code custom skills to support it.**
Below I describe what I have done and what I need Claude Code to do.

## kxue43-session-file

This is the _generic) session-management skill. It has two subcommands:
- `new` create a new boiler plate "session manifest file". The human user can then edit it by hand.
  **Read the skill definition to understand the details.**
- `load` ask Claude Code to bootstrap a new session from a session manifest file.

## kxue43-write-spec

This skill asks Claude Code to write a technical note or a spec file that captures the key ideas
and information of the current session. The spec/note file can then be referenced in the
`load` field of the front matter of a session manifest file, and then get loaded into 
a new session by `/kxue43-session-file load ...`.

## The CHANGE REQUEST

The combination of `/kxue43-session-file` and `/kxue43-write-spec` gives a _generic_ way of
assisting (automating) by session-based workflow.
**Here, I need Claude Code to implement this workflow for a _specific_ scenario – PR reviews and follow-ups.**

`.claude/skills/kxue43-pr-review/SKILL.md` is an existing skill definition that _starts_ a PR review,
but it doesn't automate the "follow-up" part – after PR author makes changes according to review comments,
validate if the comments are properly addressed by the new code and also check for newly-surfaced problems.
Also, the current skill doesn't automate the generation of session manifest files and the loading of
them in the "follow-up" session.

Make the following changes to the existing `kxue43-pr-review` skill to add the following features for me.
- The `/kxue43-pr-review` skill should have two subcommands, `start` and `followup`.
  **Checkout `.claude/skills/kxue43-session-file/SKILL.md` for how to define a skill two subcommands.**
- The `start` subcommand should be pretty much the same with the current skill, with the following changes:
  * After `start` it should take one more required positional argument `$session_file` after `$report_file`
    and before the optional `$focus_prompt`. Validation rule is the same – if not provided, stop right there
    and tell the user clearly.
  * After generating the report file, it should also generate a "session manifest file" in the same format
    created by `/kxue43-session-file new ...`, and includes both the relative paths to CWD of `$spec_file`
    and `$report_file` in the `load` field of the front matter.
    With the `goal` field, use the constant value "review if previous comments are addressed, if changes introduced new problems, and if previously missed problems can be found now".
  * When generating the report file, it should include a new field `pr_url` in the front matter and
    fill in the actual URL of the PR.
- The `followup` subcommand is a specific version of `/kxue43-session-file load ...`.
  It takes the following parameters after `followup`.
  * `session_file`: relative path to CWD of the session manifest file to load in.
  * `focus_prompt`: optional additional instructions for the follow up process.
     Treat every statement in this prompt as if it were written in bold —
     give it higher focal weight than general review guidelines when the two conflict or compete for attention.
- The `followup` subcommand first loads the session manifest file generated by `/kxue43-pr-review start ...` in the previous session,
  and then does the following specific things to follow up on the PR review process.
  * First check if the `$focus_prompt` tells if the PR author performed a force push, as force push might affect how the new changes
    should be best retrieved, and might displace the relevant code snippet a PR comment is targeted at. If there is force push,
    take these things into consideration in the following step.
  * Perform the following validations. If any fails, stop right there and tell the human user clearly.
    - Does the session manifest file and all files it references in the `load` front matter field load correctly? **Skip the session-load confirmation block as defined in `/kxue43-session-file load`.**
    - Extract `pr_url` from the front matter of the loaded report file. Then verify `git rev-parse HEAD` equals the latest commit SHA fetched from GitHub for the PR branch.
    - Fetch all review comments on the PR from GitHub. Read `changes_requested` from the front matter of the report file. Two conditions must both hold; if either fails, stop and report:
      1. Every label in `changes_requested` appears in at least one GitHub comment body (e.g. `[C1]` present in some comment text).
      2. No GitHub comment body contains a finding label (matching the pattern `[C1]`, `[M1]`, `[m1]`, etc.) that is absent from `changes_requested`.
  * Use the loaded spec file and report file to understand the PR. If Claude Code needs to use tools
    to read the code base or fetch docs, feel totally free to do that.
  * Review if the new code address the PR comments. If yes, how well?
  * Review if the new code introduce any new problems.
  * Check for problems potentially missed from the previous review sessions.
  * Finally, make a brief report to the user about comments resolution, new problems, missed problems (the three points just mentioned above).
    **Then ask the user if he would like to write a new report file and a new session manifest file for this follow-up session.**
    If user says yes, get the relative paths to CWD of the new report and session manifest files from the user's answer and write them.
    Use the same format for the report and session manifest files as `kxue43-pr-review start` would use.

**The current kxue43-pr-review skill has an existing `## Re-Review Workflow` section. Take it out of the new `/kxue43-pr-review start` subcommand and merge its contents into the new `/kxue43-pr-review followup` subcommand.**
